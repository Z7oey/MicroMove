# Codex 任务说明：优化动作匹配逻辑 + 补充坐姿/站姿标签

## 背景

当前产品里，用户的长期偏好是固定的。问题是：用户每次点击“开始运动 / 坐着动一下 / 站着动一下”之后，系统推送的动作总是一样的，体验会很无聊。

我们希望动作推荐逻辑改成：

1. 在用户偏好和当前场景允许的范围内，尽量每次推荐不一样的动作。
2. 不能为了随机而乱推，必须先满足核心匹配条件。
3. 坐姿/站姿和部位是最重要的两个匹配条件，不能错。
4. 动作数据库需要补齐“坐姿/站姿适用性”标签。
5. 未来还会加入更多只适合站姿的动作，例如小腿拉伸等，所以数据结构和匹配逻辑要能支持扩展。

---

## 一、核心问题 1：动作推荐不要每次都一样

### 当前问题

用户长期偏好固定后，每次进入同样入口，系统推荐的动作总是固定顺序或固定结果。

这会导致：

- 用户觉得动作重复、无聊。
- 即使动作本身合适，也会降低继续使用的兴趣。
- 动作库的内容没有被充分利用。

### 目标效果

在满足用户偏好和当前场景的前提下，每次尽量推荐不同动作。

例如：

- 用户选择“肩颈 + 坐姿”
- 动作库里有多个符合条件的肩颈动作
- 系统不应该每次都推同一个动作
- 应该在符合条件的动作里随机打乱，或避开最近刚做过的动作

---

## 二、核心问题 2：坐姿/站姿必须精准匹配

### 当前问题

动作数据库中可能没有清晰区分动作适合坐着做、站着做，还是坐着/站着都可以做。

但我们的产品场景是办公室微运动，用户常见入口包括：

- 坐着动一下
- 站着动一下
- 工作间隙顺手拉伸

所以动作是否适合当前姿势非常关键。

### 目标效果

当用户选择：

- “坐着动一下”时，只推荐 `坐姿` 或 `坐姿/站姿` 动作。
- “站着动一下”时，只推荐 `站姿` 或 `坐姿/站姿` 动作。
- 不要把只能坐着做的动作推荐给站姿入口。
- 不要把只能站着做的动作推荐给坐姿入口。

---

## 三、匹配优先级

请按照下面优先级修改推荐逻辑。

### 第一优先级：姿势 + 部位

姿势和部位是同等重要的核心条件。

必须同时满足：

1. 当前入口/用户选择的姿势
   - 坐姿
   - 站姿
   - 坐姿/站姿

2. 当前目标部位或用户偏好部位
   - 肩颈
   - 背部与腰部
   - 之后可能还有手腕与前臂、腿部与髋部等

### 第二优先级：空间

例如：

- 小空间
- 中空间
- 大空间

办公室场景优先小空间动作。

### 第三优先级：强度

例如：

- 温和
- 中等

目前 MVP 以温和动作为主。

### 第四优先级：可见度

例如：

- 低
- 中
- 高

办公室场景优先低可见度动作。

### 第五优先级：随机化 / 去重复

在以上条件都满足后，再进行随机打乱，尽量避免每次推荐同一个动作。

---

## 四、动作数据库需要补充 posture 标签

请检查当前新的动作数据库，确保每个动作都有明确的 `posture` 字段。

建议字段形式：

```ts
posture: "坐姿" | "站姿" | "坐姿/站姿"
```

也可以用数组形式：

```ts
postures: ["sitting", "standing"]
```

如果当前项目已有字段结构，请保持项目风格，但必须支持区分：

- 只适合坐姿
- 只适合站姿
- 坐姿/站姿都适合

---

## 五、当前已确认动作的姿势标签

请按下面规则更新当前 11 个动作。

### 肩颈动作：全部坐姿/站姿都可以

以下动作的 `posture` 都应为：

```txt

```

动作包括：

动作包括：

6. 肩后侧拉伸（左）
7. 肩后侧拉伸（右）

## 六、推荐逻辑要求

请找到当前负责动作推荐 / 匹配的代码，可能在：

```txt
src/utils/
src/lib/
src/hooks/
src/pages/
src/components/
```

也可能叫：

```txt
recommendExercise
matchExercise
getRecommendedExercise
selectExercise
generateSession
buildSession
```

请将逻辑改成以下方式。

### 1. 先筛 posture

如果用户选择“坐着动一下”：

```ts
allowed = exercise.posture === "坐姿" || exercise.posture === "坐姿/站姿"
```

如果用户选择“站着动一下”：

```ts
allowed = exercise.posture === "站姿" || exercise.posture === "坐姿/站姿"
```

如果没有明确姿势选择，则可以使用用户长期偏好或默认允许所有坐姿/站姿动作。

### 2. 再筛 category / body part

根据用户选择或偏好筛选目标部位：

- 肩颈
- 背部与腰部
- 之后可能扩展其他分类

姿势和部位都必须满足，不能只满足一个。

### 3. 再筛空间、强度、可见度

如果用户有偏好：

- space
- intensity
- visibility

请继续筛选。

如果筛选后没有结果，可以逐步放宽低优先级条件，但不要放宽姿势和部位。

可以放宽：

- 可见度
- 强度
- 空间

不要轻易放宽：

- 姿势
- 部位

### 4. 最后随机打乱

在候选动作中随机选择，而不是每次固定取第一个。

请不要使用稳定排序后固定返回第一项。

可以用：

```ts
shuffle(candidates)
```

或者：

```ts
candidates[Math.floor(Math.random() * candidates.length)]
```

### 5. 尽量避免连续重复

如果项目中已有历史记录，例如 recently completed exercises / lastExerciseId / session history，请使用它来避免重复。

推荐规则：

1. 优先排除最近 1–3 次做过的动作。
2. 如果排除后还有候选动作，就从剩余候选中随机选。
3. 如果排除后没有候选动作，再允许重复。

伪代码：

```ts
let candidates = filterByPostureAndCategoryAndPreferences(exercises, userContext)

let nonRecentCandidates = candidates.filter(
  exercise => !recentExerciseIds.includes(exercise.id)
)

let finalCandidates = nonRecentCandidates.length > 0
  ? nonRecentCandidates
  : candidates

return randomPick(finalCandidates)
```

## 七、Session 动作数量

如果一次 session 需要推荐多个动作，请遵守：

1. 先筛选符合姿势和部位的候选动作。
2. 随机打乱候选动作。
3. 按需要数量取前 N 个。
4. 同一个 session 内不要重复动作。
5. 如果候选动作数量不足，再允许补充或重复。

## 八、数据结构需要支持未来扩展

未来会加入更多只适合站姿的动作，例如：

- 小腿拉伸
- 站姿腿后侧拉伸
- 站姿髋部放松
- 靠墙胸肩拉伸

所以请确保数据结构不是写死当前 11 个动作，而是通用支持：

```ts
posture: "坐姿" | "站姿" | "坐姿/站姿"
```

或者：

```ts
postures: ["sitting", "standing"]
```

并且推荐逻辑可以自动根据 posture 过滤。

## 十、验收标准

### 数据层

- 每个动作都有 posture / postures 字段。
- 11 个已确认动作的 posture 设置正确。
- 坐式上身弯曲只能坐姿。
- 其他当前动作都是坐姿/站姿。

### 推荐逻辑

- 坐姿入口不会推荐只能站姿的动作。
- 站姿入口不会推荐只能坐姿的动作。
- 坐式上身弯曲不会出现在站姿入口。
- 肩颈动作在坐姿和站姿入口都可以出现。
- 背部与腰部动作中，除坐式上身弯曲外，其他坐姿/站姿都可以出现。

### 随机化

- 同样偏好下，多次点击开始，不应该总是同一个动作。
- 在候选动作足够多时，最近刚做过的动作应尽量不连续出现。
- 同一个 session 内不要重复动作。

### 页面

- 动作库仍然正常显示。
- 动作详情页仍然正常显示。
- 图片路径不被破坏。
- 旧动作没有重新出现。

### 构建

完成后运行：

```bash
npm run build
```

并报告：

1. 修改了哪些文件。
2. 动作数据库在哪里。
3. 推荐逻辑在哪里。
4. 是否 build 通过。
5. 是否已经验证坐姿/站姿入口的推荐结果不同且不总是重复。

---

## 十一、重要限制

请严格遵守：

1. 不要重新设计 UI。
2. 不要改动动作图片。
3. 不要重新命名已确认动作。
4. 不要删除现有 11 个动作。
5. 不要把姿势和部位作为低优先级条件，它们是核心匹配条件。
6. 不要为了随机而忽略 posture。
7. 不要为了随机而忽略 category / body part。
8. 如果发现当前入口没有姿势信息，请补充传参，而不是随便默认。
9. 如果发现数据库字段结构和本文档不一致，请按当前代码风格适配，但功能要求必须满足。
10. 如果发现无法判断用户当前姿势或部位，请暂停并说明，不要自行猜测。

---

## 一句话总结

请为动作数据库补齐坐姿/站姿适用性标签，并重写动作推荐逻辑：先精准匹配姿势和部位，再匹配空间、强度、可见度，最后在候选动作中随机选择并尽量避免连续重复。目标是让用户在符合偏好的前提下，每次看到的动作尽量不一样，同时不破坏坐姿/站姿和部位匹配的准确性。
