# UI / Interaction Revision 1：办公室场景与事件绑定

本轮只修改用户交互体验、页面信息结构和文案逻辑。不要重构核心 session 生成逻辑，不要改 exercises 数据结构，不要新增后端、登录、数据库或真实系统推送。

## 1. 本版用户画像固定为办公室人群

MVP v1 只针对办公室人群，不再泛化成所有微运动用户。

这一版默认用户画像是：长时间坐在电脑前工作的办公室用户。他们需要的是低强度、小幅度、办公室友好、不尴尬、可以在工作间隙顺手完成的动作。

因此页面文案和动作推荐逻辑要更贴近办公室事件，而不是泛泛地问“你哪里痛”。

### 文案方向

不要太依赖疼痛表达，例如不要频繁说：

- 你是不是肩颈酸痛？
- 你哪里不舒服？
- 身体在向你求救

更推荐使用办公室事件和状态切换表达，例如：

- 刚切完一个任务，肩颈也重启一下。
- 回座前 30 秒，身体先回个电。
- 送完文件回来，顺手松一下肩。
- 刚看完一轮电脑，给身体换个档。
- 开会前先松一下，别带着僵硬进去。
- 接完水回来，顺手活动一下手腕。
- 打完一段字，手腕也休息 30 秒。

整体语气：实用、轻松、有一点打工人共情，但不要太温柔、不要太治愈、不要太健身挑战。

## 2. 首页增加“坐着动一下 / 站着动一下”大分类选择

用户完成 onboarding 后，进入主页面时，应先看到两个主要入口：

- 坐着动一下
- 站着动一下

这两个入口作为主页面的大 tab / 大卡片选择。

### 坐着动一下

适用场景：

- 连续办公
- 看电脑太久
- 看手机太久
- 懒得起身
- 正在工位上
- 不想动作太明显

文案方向示例：

- 不想起身也可以，先坐着松一下。
- 连续看屏幕太久，先把肩颈放下来。
- 懒得站起来的时候，也能动 30 秒。
- 坐着也能换个状态。

动作筛选倾向：

- 优先小空间
- 优先坐姿
- 优先低可见度
- 优先温和动作

### 站着动一下

适用场景：

- 接水
- 送文件
- 签字
- 开会前后
- 上厕所回来
- 打印
- 离开工位
- 找同事回来

文案方向示例：

- 都站起来了，顺手松一下。
- 接水回来，给身体回个电。
- 开会前 30 秒，把状态拉回来。
- 打印等一下，顺便活动一下腿和腰。
- 回座前，先别急着坐下。

动作筛选倾向：

- 可使用站姿动作
- 允许中空间动作
- 仍然保持办公室友好
- 不要推荐夸张、健身感强或尴尬动作

## 3. 动作页面增强办公室 event 绑定

Exercise Player 页面中，动作说明之外，可以增加一条短的滚动/随机场景文案，用来把动作和办公室事件绑定。

这些文案不影响动作逻辑，只作为页面提示语展示。

### 可用事件词库

- 会议前
- 会议后
- 接水回来
- 上厕所回来
- 拿东西回来
- 找同事回来
- 打印等待时
- 回到工位前
- 刚切完任务
- 刚写完一段内容
- 刚回完一轮消息
- 刚看完一段电脑

### 示例文案

- 刚切完任务，顺手把肩颈也切回来。
- 回到工位前，先给身体 30 秒。
- 接水回来，别急着坐下。
- 打印等一下，刚好松松腿。
- 开会前，把肩膀从耳边放下来。
- 刚回完消息，手腕也休息一下。
- 看完一轮屏幕，眼睛和脖子都该换个档。
- 坐久了，不用大动，先松一点。

要求：

- 文案简短。
- 不要矫情。
- 不要像健康科普。
- 不要像健身挑战。
- 不要影响倒计时和动作说明的可读性。

## 4. Reminder 功能调整：MVP 不做真实系统提醒，但允许用户设置 App 内闹钟

MVP v1 暂时不做真实系统推送通知，不做手机锁屏提醒，不做后台通知，不做 service worker / push notification。

但允许用户在 app/web 内设置一个“提醒闹钟”。

### 功能定义

用户可以在页面中设置一个提醒时间或间隔，例如：

- 45 分钟后提醒
- 60 分钟后提醒
- 90 分钟后提醒
- 自定义一个时间点

当网页仍然打开时，到时间后在 app 内弹出提醒弹窗。

提醒弹窗文案示例：

- 该松一下了。
- 坐太久了，换个档。
- 刚好停一下，做一组 30 秒动作。
- 起来前后都行，先动一下。

按钮：

- 开始
- 稍后提醒
- 今天先不提醒

### 重要限制说明

如果目标是：

用户设定一个固定时间（例如每天 10:30、14:00），到了时间后手机能够收到提醒，

那么这是可以做到的，但需要实现真正的系统通知能力，而不是仅限于网页内弹窗。

请在代码和 UI 中支持两种模式：

1. App 内提醒（默认）

- 当网页保持打开时，到时间后在页面内弹出提醒。
- 不依赖系统通知权限。

2. 系统通知（推荐）

- 用户授权浏览器通知权限后，可以在设定时间发送系统通知。
- 通知会出现在手机通知栏，而不仅仅是网页页面内。
- UI 中应明确提示用户需要开启通知权限。

需要注意：

- iOS Safari、Android 浏览器以及不同设备对后台通知支持存在差异。
- 如果采用 PWA + Push Notification 或原生 App，可以获得更稳定的提醒体验。
- 在纯网页环境下，浏览器关闭、系统省电策略或后台限制可能影响提醒可靠性。
- 不要向用户承诺“100% 必达”，但应尽可能提供接近手机提醒的体验。

UI 文案建议：

- 开启系统提醒（推荐）
- 允许通知权限后，可在设定时间收到手机通知。
- 如果通知权限未开启，仍可使用网页内提醒。

## 5. 不要改动的内容

本轮不要改以下内容：

- 不改每个动作 30 秒。
- 不改 session 时长硬约束。
- 不改 completed / abandoned session 记录逻辑。
- 不改 CSV 导出逻辑。
- 不改 localStorage 存储方式。
- 不新增后端。
- 不新增登录。
- 不新增社区、排行榜、AI 教练或穿戴设备连接。
- 不删除动作库分类。
- 不删除 Profile 里的导出使用记录。

## 6. 本轮验收标准

完成后应满足：

1. 首页能看到“坐着动一下 / 站着动一下”两个主入口。
2. 用户选择坐姿入口时，优先生成坐姿、小空间、低可见度动作。
3. 用户选择站姿入口时，可以生成站姿和中空间动作，但仍保持办公室友好。
4. 页面文案明显更贴近办公室事件，而不是泛泛询问疼痛。
5. 动作播放器中出现简短办公室事件提示语。
6. Reminder 不再表现为真实系统推送，而是明确为网页内提醒/闹钟。
7. 原有 onboarding、session、completion、Profile、CSV 导出功能不被破坏。
8. 修改完成后运行 npm run build。
